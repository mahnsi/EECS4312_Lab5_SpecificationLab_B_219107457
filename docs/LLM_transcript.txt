Your current tests are mostly checking *per-request* constraints. The hidden trap here is global aggregation. If three requests each ask for 4 CPUs and you only have 10 total, the allocation is infeasible — even though no single request exceeds capacity.

That’s the classic “locally valid, globally impossible” bug. Your current implementation would incorrectly return `True` in that case.

Here are three strong, non-redundant tests that probe edge cases and deeper correctness:

```python
def test_total_aggregate_exceeds_capacity():
    # Aggregate Exceeds Capacity
    # Constraint: total demand across all requests must not exceed total capacity
    # Reason: catches implementations that only check per-request limits
    resources = {'cpu': 10}
    requests = [{'cpu': 4}, {'cpu': 4}, {'cpu': 4}]  # total = 12 > 10
    assert is_allocation_feasible(resources, requests) is False


def test_multi_resource_aggregate_exceeds_one_dimension():
    # Multi-Resource Aggregate Overload (single dimension)
    # Constraint: total demand per resource must be respected independently
    # Reason: ensure aggregation happens per-resource, not per-request
    resources = {'cpu': 10, 'mem': 20}
    requests = [
        {'cpu': 3, 'mem': 10},
        {'cpu': 3, 'mem': 5},
        {'cpu': 3, 'mem': 6}  # mem total = 21 > 20
    ]
    assert is_allocation_feasible(resources, requests) is False


def test_request_with_unmentioned_resource_defaults_to_zero():
    # Partial Resource Usage
    # Constraint: resources not mentioned in a request are treated as 0
    # Reason: ensure function does not incorrectly require all resources per request
    resources = {'cpu': 5, 'mem': 10}
    requests = [
        {'cpu': 3},
        {'mem': 8}
    ]
    assert is_allocation_feasible(resources, requests) is True
```

Why these matter:

* The first two expose the conceptual flaw in your current implementation (it does not aggregate usage).
* The third checks correctness without being redundant — it verifies that sparse requests behave properly.

If you run these against your current function, at least the first two will fail. That’s not bad news — it means your tests are doing their job. A test suite that can’t break your code isn’t a test suite, it’s a cheerleader.

This is a neat microcosm of distributed systems resource accounting: local checks are easy; global invariants are where correctness lives.

Yes — one category is still uncovered, and it matters if you want true robustness rather than just “passes the happy path.”

You don’t currently test **invalid numeric values**.

Right now your function silently accepts negative numbers. That creates logical nonsense like:

* A request asking for `-3` CPUs.
* A resource capacity of `-10`.

From a modeling standpoint, that’s equivalent to saying someone is *producing* CPU by requesting it. That’s thermodynamics crying softly in the corner.

If your spec assumes non-negative quantities (which almost all allocation models do), you should test that negative values either:

1. Raise an error, or
2. Cause infeasibility.

Example test:

```python
def test_negative_request_amount():
    # Negative Request Amount
    # Constraint: resource demands must be non-negative
    # Reason: prevent logically invalid allocations
    resources = {'cpu': 10}
    requests = [{'cpu': -3}]
    with pytest.raises(ValueError):
        is_allocation_feasible(resources, requests)
```

You might also consider negative capacities:

```python
def test_negative_resource_capacity():
    # Negative Resource Capacity
    # Constraint: available resources must be non-negative
    # Reason: invalid system state should not be treated as feasible
    resources = {'cpu': -5}
    requests = [{'cpu': 1}]
    with pytest.raises(ValueError):
        is_allocation_feasible(resources, requests)
```

Whether you enforce this depends on the lab specification — but if the model represents real resources, non-negativity is part of the mathematical contract.

Beyond that, your coverage is conceptually solid once aggregation is tested. The only remaining frontier would be floating-point edge cases (e.g., 0.1 + 0.2 precision issues), but that’s rarely required unless the assignment explicitly mentions floats.

The deeper lesson: coverage isn’t about quantity of tests. It’s about attacking invariants. Resource allocation has three invariants:

* Structural validity
* Non-negativity
* Global capacity constraints

Break those, and your model becomes science fiction instead of accounting.
